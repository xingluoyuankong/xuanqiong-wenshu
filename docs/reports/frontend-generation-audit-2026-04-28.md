## 3. 页面级深度审计

### 3.1 首页 `/`

**实际作用**

- 应该作为用户进入系统后的总控制台。
- 应该让用户一眼知道：新建灵感、继续小说、管理配置、查看运行状态。

**当前问题**

1. 原首页设计风格与后续页面不统一；此前部分入口含义混乱。
2. 首页承担了“入口解释”但没有承担“系统状态看板”。
3. 最近项目只显示基础进度，不显示“是否卡住”“是否待确认”“是否失败”。
4. 没有清晰展示“正在生成中的章节”和“需要用户确认的版本”。

**已修正**

- 已改成创作流水线式首页。
- 已修正系统配置/LLM 配置入口。

**完全重构方向**

- 首页应成为「创作驾驶舱」：
  - 左侧：继续写作卡片。
  - 中部：创作流水线。
  - 右侧：后台任务、LLM 状态、失败重试提醒。
  - 底部：最近项目矩阵。
- 首页不应只是按钮集合，要成为“今日创作状态总览”。

### 3.2 小说项目 `/workspace`

**实际作用**

- 管理全部小说项目。
- 进入灵感续写或写作台。
- 创建/删除项目。

**测试依据**

- `GET /api/novels` 通过。
- 数据显示当前列表存在多个项目，说明项目列表与读取链路可正常工作。

**当前问题**

1. 项目卡片没有足够区分：
   - 蓝图未完成。
   - 正在生成。
   - 待确认。
   - 失败。
   - 已完成。
2. 删除是高风险动作，UI 应更严格做二次确认和影响说明。
3. 项目列表没有过滤器：生成中、失败、未完成灵感、已成稿。
4. 没有项目健康评分：章节缺失、失败任务、乱码标题、空章节。

**完全重构方向**

- 项目列表改为「项目健康看板」：
  - 状态 Badge：灵感中 / 蓝图完成 / 写作中 / 有失败 / 待确认。
  - 进度环：已完成章节 / 总章节。
  - 风险提示：生成卡住、乱码标题、章节 0 字。
  - 快捷动作：继续、查看日志、修复、导出。

### 3.3 灵感模式 `/inspiration`

**实际作用**

- 用对话收集创意。
- 生成结构化蓝图。
- 保存蓝图后进入写作台。

**真实测试**

创建临时项目后调用：

```text
POST /api/novels/{audit_id}/concept/converse
结果：200
耗时：10749ms
```

返回了可用的 `ai_message`、`ui_control`、`conversation_state`。

**发现的严重问题**

1. **首轮对话没有吸收用户输入。**  
   我传入了具体设定，但返回内容仍像“请告诉我你的灵感是什么”的首问。这说明当前概念对话 prompt 或历史构造没有正确利用首轮 user_input，或者 concept prompt 强制走固定开场。

2. **蓝图生成是同步长请求，实测 520 秒超时。**

```text
POST /api/novels/{audit_id}/blueprint/generate
结果：客户端 520 秒 ReadTimeout
```

这对前端是致命体验问题：用户等 8 分钟后仍可能不知道任务是否还在后台继续。

3. **蓝图生成失败后没有任务化恢复机制。**
   - 没有 job id。
   - 没有进度查询。
   - 没有“继续等待/取消/降级生成/使用简版蓝图”的明确路径。

4. **灵感模式 UI 过重。**
   `InspirationMode.vue` 约 1134 行，布局、状态机、业务调用、恢复逻辑耦合。

**完全重构方向**

- 灵感模式改为四段式：
  1. 火种输入：一句话/片段/人物/世界观。
  2. 结构追问：题材、主角、冲突、主题、篇幅。
  3. 蓝图生成任务：必须异步化，有 job id 和进度。
  4. 蓝图编辑台：可视化编辑人物/章节/伏笔。
- 蓝图生成必须支持：
  - 快速蓝图（30-60 秒）
  - 深度蓝图（后台任务）
  - 失败降级：用已收集信息生成手动可编辑草案。

### 3.4 写作台 `/novel/:id`

**实际作用**

- 项目的核心生产界面。
- 承担章节生成、任务进度、版本选择、评审、编辑、导出。

**代码现状**

| 文件 | 行数 | 问题 |
|---|---:|---|
| `WritingDesk.vue` | 2268 | 过大，页面状态、任务、弹窗、业务全部耦合 |
| `WDWorkspace.vue` | 1074 | 工作区继续过重 |
| `ChapterGenerating.vue` | 940 | 生成状态展示复杂且测试失败 |
| `VersionSelector.vue` | 620 | 候选版本逻辑应独立成 feature |

**真实章节生成测试**

我创建临时项目，保存蓝图后调用：

```text
POST /api/writer/novels/{audit_id}/chapters/generate
请求 target_word_count=700, min_word_count=350
结果：200，后台任务创建成功
```

但日志显示：

```text
request_target=700 request_min=350 effective_target=3000 effective_min=3000
```

**严重问题**

1. **用户传入字数被强制改成 3000。**  
   这解释了为什么短篇测试也会变成重任务。前端即使提供“短篇/目标字数”，后端也不尊重，用户会觉得功能失效。

2. **章节任务卡在 `generate_variants`。**  
   轮询超过 2 分钟仍停在：

```text
generation_status=generating
progress_stage=generate_variants
progress_percent=34
word_count=0
```

3. **状态没有过期/卡住判定闭环。**  
   前端有“可能卡住”的展示，但后端没有自动转 failed，也没有 last_error_summary。

4. **生成流程太重。**  
   即使短章节，也进入 longform preset，且 target/min 被提升到 3000。

5. **前端测试与实现不一致。**
   `chapterGeneration.spec.ts` 中 5 个失败，说明阶段标签、摘要、状态归一化已经失控。

**完全重构方向**

- 写作台必须重做成「三栏专业写作 IDE」：
  - 左栏：章节导航 + 状态过滤。
  - 中栏：正文/候选版本/大纲编辑。
  - 右栏：生成控制台 + 评审 + 记忆/伏笔/知识图谱。
- 任务状态必须成为第一等对象：
  - `task_id`
  - `stage`
  - `progress`
  - `started_at`
  - `last_heartbeat`
  - `can_cancel`
  - `can_retry`
  - `stale_reason`
- 生成模式拆分：
  - 快速草稿：500-1200 字，单版本。
  - 标准章节：2500-4000 字。
  - 精修章节：多版本 + 评审 + 优化。

## 4. 生成小说质量抽检

数据库中：

| 数据 | 数量 |
|---|---:|
| 小说项目 | 40+，前端当前用户列表为 7-9 个 |
| 章节记录 | 35 |
| 章节大纲 | 254 |
| 章节版本 | 64 |

抽检代表项目：某个长篇样本项目（名称已脱敏）

### 优点

1. 氛围塑造强。
2. 类型感明确：赛博、记忆、悬疑、克苏鲁式未知。
3. 章节标题有辨识度。

### 缺陷

1. **描写堆叠过多。**  
   环境、气味、墙壁、灯光、腐朽意象反复出现，信息推进慢。

2. **章节长度不稳定。**  
   有章节 10000+ 字，也有 0 字/失败/生成中章节。

3. **语言洁净度不足。**  
   抽检出现英文夹杂，如 `sockets`。

4. **节奏缺少硬约束。**  
   很多段落是氛围推进，不是事件推进。

5. **连续性仍需自动校验。**  
   人物状态、伏笔回收、前章依据需要成为生成前硬约束。

### 小说生成优化方向

- 增加“章节目标检查器”：
  - 本章必须发生什么事件。
  - 本章必须推进什么关系。
  - 本章必须新增/回收什么伏笔。
  - 本章必须留下什么钩子。
- 增加“重复意象检查器”。
- 增加“中文洁净度检查器”。
- 增加“目标字数硬约束”，后端不得擅自把 700 改成 3000。

## 6. 完全前端重构目标

你要的不是“修修补补”，而应是一次完整的产品级重构。

### 6.1 新视觉方向

建议视觉主题：**「玄墨星图 · 创作驾驶舱」**

关键词：

- 深色玄墨背景
- 米白纸张阅读区
- 青蓝/紫色星图光效
- 低饱和金色强调
- 卡片玻璃质感
- 写作区保持安静，不花哨

### 6.2 设计系统

```text
Design System
├─ 色彩
```

## 7. 前端重构执行计划

### 阶段 0：先止血

1. 修复 8 个失败测试。
2. 修复所有用户可见乱码。
3. 章节生成状态超过阈值自动显示“疑似卡住”并给出取消/重试。
4. LLM 模型列表区分 masked key 与真实 key。
5. 蓝图生成按钮改为异步任务 UI，即使后端暂未任务化，也要前端有超时保护。

### 阶段 1：设计系统落地

1. 新建 `shared/ui`。
2. 建立 tokens：颜色、间距、圆角、阴影、字体、动效。
3. 全局 CSS 清理，禁止页面散落大量不统一样式。
4. 建立 Story-like 组件展示页或内部 `/dev/ui`。

### 阶段 2：首页 + 项目页重构

1. 首页改创作驾驶舱。
2. 项目页改项目健康看板。
3. 新增项目过滤与风险标记。
4. 最近项目显示生成中/待确认/失败。

### 阶段 3：灵感模式重构

1. 抽 `useInspirationSession`。
2. 抽 `CollectedIdeaPanel`。
3. 抽 `BlueprintGenerationTask`。
4. 蓝图确认改为可视化编辑画布。
5. 支持快速蓝图/深度蓝图。

### 阶段 4：写作台重构

1. 拆 `WritingDesk.vue`。
2. 建立 task store。
3. 建立 chapter store。
4. 建立 version store。
5. 建立 review store。
6. 写作台三栏布局。
7. 所有生成动作进入统一任务控制台。

### 阶段 5：故事圣经与分析重构

1. 小说详情重命名为故事圣经。
2. 情绪曲线、伏笔、线索、知识图谱统一视觉语言。
3. 每个问题都能转化为写作约束或修复任务。

### 阶段 6：管理台重构

1. 管理台改运维驾驶舱。
2. 日志按 task_id 聚合。
3. 错误码、root cause、修复建议可视化。

## 8. 后端配合改造

前端要“完美”，后端必须配合，否则 UI 再好也会被生成链路拖垮。

### 必改

1. 蓝图生成任务化：
   - `POST /blueprint/generate-task`
   - `GET /blueprint/tasks/{task_id}`
   - `POST /blueprint/tasks/{task_id}/cancel`

2. 章节生成尊重字数：
   - 不得强制把 700 改 3000。
   - 如果 preset 要覆盖，接口必须返回 warning。

3. 任务心跳：
   - `last_heartbeat_at`
   - `stale_after_seconds`
   - `stale_reason`

4. 日志 UTF-8 修复。

5. 生成失败必须回写：
   - `generation_status=failed`
   - `last_error_summary`
   - `retryable`
   - `suggested_action`

## 9. 下一步建议

如果继续执行，我建议按以下顺序动手：

1. **先修 P0 后端/状态问题**：蓝图超时、章节卡住、字数不生效。
2. **建立前端设计系统骨架**：tokens + AppShell + 三栏布局。
3. **重写首页和项目页**：这两页风险最低，能确定整体视觉。
4. **重写灵感模式**：改成四阶段。
5. **重写写作台**：最大工程，必须拆组件和 store。
